# イベント現場用リアルタイム情報共有システム 設計書 (Step 1)

## 1. プロジェクト概要
イベント現場のスタッフが、スマホから迅速に状況（クレーム、医事対応、混雑状況等）を報告し、本部および全スタッフでリアルタイムに情報を同期・管理するためのWebアプリケーション。

### 主な目標
- **入力負荷の最小化**: カテゴリ選択による入力補助とステータスのプリセット。
- **情報の透明化**: 全スタッフが最新状況を把握でき、かつ編集履歴を確認可能にする。
- **デバイス最適化**: スマホ（簡易リスト・入力重視）とPC（全体俯瞰・管理重視）の出し分け。

## 2. 技術スタック
- **バックエンド**: Python (FastAPI), Uvicorn
- **フロントエンド**: HTML5, CSS3 (Vanilla), JavaScript (ES6+, Framework-less)
- **通信**: WebSocket (Real-time sync)
- **データ保持 (暫定)**: インメモリ（サーバー起動中のみ保持）
  - ※Step 2以降でSQLite/PostgreSQLなどの永続化DBへ移行予定。
- **クライアント側名前保持**: localStorage

## 3. 主要機能とUI/UX設計

### 3.1 報告入力 (スマホ重視)
- **報告者名**: 初回入力時にlocalStorageに保存し、次回以降自動補完。
- **入力補助 (ドリルダウン形式)**:
  - カテゴリ選択（例：医事対応、機材トラブル、クレーム、混雑）
  - カテゴリに応じた詳細項目（例：医事対応の場合 → 発熱、怪我、熱中症、その他）
  - 自由記述テキストエリア。
- **送信**: 送信ボタンタップでWebSocket経由で即座にブロードキャスト。

### 3.2 状況一覧・詳細
- **スマホ版UI**: 
  - 1項目をカード形式で簡潔に表示（時刻、送信者、カテゴリ、ステータスのみ）。
  - タップで詳細画面（本文、履歴）へ。
- **PC版UI**:
  - 表形式（Table）で全項目を一画面に表示。
  - 行をクリック、または編集ボタンでその場でステータス更新や内容修正が可能。

### 3.3 編集・履歴管理
- **履歴構造**:
  - 各報告は一意の `report_id` を持つ。
  - 編集のたびに `version` をインクリメントした新しいレコードを作成（イミュータブル形式）。
  - 画面上には常に最新 `version` を表示。
  - 詳細画面で「いつ、誰が、何を」変更したかの履歴をタイムライン表示。
- **ステータス・プリセット**:
  - 未対応 / 対応中 / 完了
  - 特定カテゴリ用（例：医事の場合「医務室へアテンド中」「救急搬送済み」など）

## 4. システムアーキテクチャ

### 4.1 FastAPI ルーティング
- `/`: メインアプリケーション（レスポンシブHTML）
- `/ws`: WebSocket エンドポイント
- `/api/v1/...`: 将来的なREST API拡張用（認証対応予定）

### 4.2 WebSocket メッセージ形式 (JSON)
```json
{
    "type": "new_report" | "update_report",
    "data": {
        "report_id": "uuid",
        "version": 1,
        "sender": "名前",
        "category": "カテゴリ名",
        "detail": "詳細項目",
        "message": "本文",
        "status": "ステータス",
        "timestamp": "ISO時刻"
    }
}
```

## 5. 実装に向けたステップ (Step 1)
1. `app.py` の基本実装（FastAPI + WebSocket管理クラス）。
2. 静的ファイル（HTML/CSS/JS）の配信設定。
3. `index.html` (JSによる動的レンダリング) の実装。
4. 入力補助（ドリルダウン）とlocalStorageの連携。
5. リアルタイム同期の疎通確認。
